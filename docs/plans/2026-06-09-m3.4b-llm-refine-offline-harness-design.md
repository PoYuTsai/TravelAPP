# M3.4b — LLM Refine Offline Harness（設計）

- 日期：2026-06-09
- Branch：`codex/line-oa-agent-mvp`
- 前置：M3.4a（live-masked retrieval theme-signal）已收。M3.3a–e（lint / composer / change composer / operator preview）已綠。

## 1. 目標與邊界

在既有 pipeline（`composeCustomerChange` → deterministic `customer_itinerary_v1` draft → lint gate）**後面**加一層「LLM refine」。LLM 是**不可信的潤飾器**：可改善客人版語氣，但**不得**改任何 itinerary fact。refine 後的輸出一定要再回頭過三道確定性閘門；任一不過即 fail-closed、**不採用** refined output、退回 deterministic draft。

本刀**只能**：
- fixture-only、offline harness（注入 fake `RefineDraftSource`，不接真 LLM）
- 只 refine itinerary draft（**不**處理 `customerExplanation`，留下一刀）
- refined output 必過 `lintCustomerItinerary`
- refined output 必過 `structuralDiffGuard`（新增）
- refined output 必過 leak / forbidden-terms guard
- 任一 guard fail → fail-closed，不採用 refinedDraft
- 可比較 deterministic draft vs refined draft

本刀**不**：不接 LINE、不翻 partner/customer gate、不寫 Sanity、不產正式 quote URL、不讓 LLM 繞過 deterministic/lint gate。

## 2. 可潤表面（v1 故意最小化）

決策（Eric 2026-06-09）：v1 把可潤範圍縮到最小。

**逐字凍結**（任何改動 → 結構漂移 → fail-closed）：
- 日期行 `📅 日期：…`、人數行 `👨‍👩‍👧‍👦 人數：…`
- Day 數與順序
- 每個 `・` 活動行、`午餐：`/`晚餐：` 行、`・住宿：` 行
- final day morning transfer 仍不得出現 lunch/dinner/lodging

**允許微調**：
- 標題行 `<X套餐訂製> headerTitle`
- 第一個 Day 之前的開場白 / 收尾語等非 itinerary-fact 客人語氣
- Day 標題行（`Day N｜title`）：**只**允許標點/空白微調；不可新增/刪除景點名

> 設計理由：這一刀的產出不是「更漂亮的行程」，而是「可信的不可信潤飾器框架」。v1 先把可潤表面壓到最小，證明三道閘門擋得住漂移/洩漏/壞格式；放寬到「景點名一致但允許加溫度註解」的 fuzzy guard 留下一刀。

## 3. 架構

### 3.1 seam：`RefineDraftSource`（注入點）

```ts
export interface RefineDraftRequest {
  deterministicDraft: string
  constraints: CustomerItineraryConstraints
}
export type RefineDraftSource =
  (req: RefineDraftRequest) => string | Promise<string>
```

與 M3.2 `PartnerRagDraftSource` 同 pattern：注入點是 source，守門永遠是確定性程式碼。offline harness 注入 fixture fake source。真 LLM adapter 之後可不改 harness 直接 slot-in。

### 3.2 harness：`refineCustomerItineraryDraft`（async）

輸入：`{ deterministicDraft, constraints, source }`。前置條件：`deterministicDraft` 已過 lint（由 composer 保證）。

流程：
1. `candidate = await source({ deterministicDraft, constraints })`（try-catch；throw/非字串/空字串 → `source_error` 直接 fail-closed）
2. guard A：`lintCustomerItinerary(candidate, constraints)` 必須 ok（無 error）
3. guard B：`structuralDiffGuard(deterministicDraft, candidate, constraints)` 必須無 issue
4. guard C：leak / forbidden-terms guard，candidate 不得含內部詞彙
5. 三道全過 → 採用 refined；否則 fail-closed 退回 deterministic

### 3.3 結果型別

```ts
export interface RefineResult {
  draft: string                    // 實際採用：refined 或 deterministic（永遠非 null）
  deterministicDraft: string
  refinedDraft: string | null      // 未採用時為 null
  used: 'refined' | 'deterministic'
  rejectionReasons: string[]       // used==='deterministic' 時非空
  structuralIssues: StructuralDiffIssue[]
  lintIssues: ItineraryLintIssue[]
  leakHits: string[]
}
```

不變式：`used==='refined' ↔ refinedDraft!==null && rejectionReasons.length===0`；`draft` 永遠是可用的客人版（fail-closed 時 = deterministic）。

## 4. structuralDiffGuard（核心）

`extractDraftStructure(text)` 把 v1 文字抽成：

```ts
interface DayStructure {
  day: number
  heading: string          // 'Day N｜title' 原文
  activities: string[]     // ・ 活動行（排除 ・住宿）
  lunch: string | null
  dinner: string | null
  lodging: string | null
}
interface DraftStructure {
  dateLine: string         // 📅 日期：…
  partyLine: string        // 👨‍👩‍👧‍👦 人數：…
  days: DayStructure[]
}
```

`structuralDiffGuard(det, ref, constraints)` 逐項比對，產出 `StructuralDiffIssue[]`：
- `date_line_changed` / `party_line_changed`：日期/人數行非逐字相同
- `day_count_changed` / `day_order_changed`：Day 數或 day 序列不同
- `day_title_changed`：Day 標題行**去空白+去標點**後仍不同（允許標點/空白微調；新增/刪除景點名即觸發）
- `activities_changed`：某日活動行陣列非逐字相同（順序敏感）
- `lunch_changed` / `dinner_changed` / `lodging_changed`：某日餐/住宿內容非逐字相同
- `final_day_meal_or_lodging`：final day morning transfer 下 refined 出現 lunch/dinner/lodging（與 deterministic 一致性比對的顯式再保險）

Day 標題正規化：移除所有空白與標點集合（`、，,／/・·…（）()「」 空白`）後比對剩餘字元。

## 5. leak / forbidden-terms guard

抽出共用單一事實來源 `customer-facing-forbidden-terms.ts`：
- `FORBIDDEN_CUSTOMER_TERMS`（小寫 latin：internal/retrieval/source/themetag/mobilityfriendly/operator/lint/fail-closed/system…）
- `FORBIDDEN_CUSTOMER_CJK`（內部/候選/白名單/代入/來源 case/否決/decision…）

`scanCustomerForbiddenTerms(text)` → `string[]` 命中清單。M3.3e explanation test 改 import 同一份清單（去重複定義）。refine guard 對 refined draft 套同一份；命中即 fail-closed。

## 6. 比較 deterministic vs refined

harness 回傳已含 `deterministicDraft` / `refinedDraft` / `used` / `*Issues`，足以比較。另加輕量 `summarizeRefineOutcome(result)` → operator-facing 文字（used、rejectionReasons、各 guard issue 數），供 operator 出手前 trace（不接 LINE、不印 PII 以外的內部 id）。
