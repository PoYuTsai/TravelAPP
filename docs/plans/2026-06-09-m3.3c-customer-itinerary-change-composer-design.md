# M3.3c — Deterministic Customer-Change Composer 設計

- 日期：2026-06-09
- Branch：`codex/line-oa-agent-mvp`
- 前置：M3.3a lint（`customer-itinerary-lint.ts`）、M3.3b composer skeleton（`customer-itinerary-composer.ts`）
- 定位：**feasibility / advice layer**。模擬 Eric 的專業排程流程，不是「客人說什麼就改什麼」。

## 目標修正（vs 原 M3.3c）

原規劃 M3.3c = retrieval-driven 建議 + 末日餐宿 auto-fix + LLM refine。
本設計改為 **deterministic customer-change composer**：

客人變動需求 → deterministic 合理性判斷 → 直接產出較舒服、可執行的
`customer_itinerary_v1` → 附 `customerExplanation` / `operatorNotes`。

零 LLM、零 RAG live、零 CLI、不接 LINE / Sanity、不翻 gate。

## §1 架構與輸出契約

**新檔** `src/lib/line-agent/notion/customer-itinerary-change-composer.ts`。
**複用** M3.3b `renderCustomerItinerary` + M3.3a `lintCustomerItinerary` 當最後品質閘。
change layer 只負責 transform + decision classification；M3.3b composer / lint gate 最後把關。

主函式：

```ts
composeCustomerChange(input: {
  base: ComposeCustomerItineraryInput     // 李家 golden-like 結構化基底
  changes: CustomerChangeRequest          // 結構化客人變動需求
  retrievalCases?: RetrievalCaseRef[]     // 只當「替代景點白名單」，不可自由 hallucinate
}): CustomerChangeResult
```

輸出契約：

```ts
interface CustomerChangeResult {
  ok: boolean                  // 定義為 (draft !== null)，即是否有客服可用草稿
  draft: string | null         // customer_itinerary_v1；硬衝突時 null（fail-closed）
  customerExplanation: string  // 給客服直接貼給客人的幾句話
  operatorNotes: string[]      // 只給內部看：每筆調整/否決理由
  issues: ItineraryLintIssue[] // fail-closed 時的殘餘 lint 違規
}
```

三段式流水線：

1. `applyChanges(base, changes, retrievalCases)` → 純函式，回 `{ adjustedInput, decisions[] }`。
   `decisions` 記每筆變更為 `applied` / `adjusted` / `declined` + 理由。
2. 把 `decisions` 投影成 `customerExplanation`（客人語氣、過濾內部詞）與 `operatorNotes`（內部理由）。
3. `composeCustomerItineraryDraft(adjustedInput)`（M3.3b 既有閘）→ render → lint。

### 不可違反的契約細則

- `ok === (draft !== null)`。
- lint **warning 不阻擋** draft；lint **error 才 fail-closed**（draft=null）。
- fail-closed 時 draft=null，但 `customerExplanation` / `operatorNotes` **仍要回**，方便客服專業回覆客人。
- `customerExplanation` **禁止**出現 `internal` / `lint` / `fail-closed` / `system` 等詞；只用客人聽得懂的旅遊語言。
- draft 永遠走「客人乾淨版」，所有調整理由走 explanation / operatorNotes 旁路 → 天然滿足「draft 不輸出過多 internal/operator notes」。
- `retrievalCases` 只能當替代景點白名單；找不到就泛稱「較輕鬆的替代景點」，**絕不**杜撰新景點名。

## §2 CustomerChangeRequest schema 與 feasibility 規則

```ts
interface CustomerChangeRequest {
  knownFlight?: CustomerItineraryKnownFlight   // 已知/變更航班
  mobilityDowngrade?: CustomerItineraryMobility// 長輩體力更差
  sameLodgingAllTrip?: { stayArea: string }    // 堅持全程同住
  lodgingChangeRequests?: Array<{              // 住宿變更獨立欄位（不塞 addActivities）
    day: number; lodging: string
  }>
  finalDayMorningTransfer?: { time: string }   // 末日早上送機
  addActivities?: Array<{ day: number; activity: string }>
  removeActivities?: Array<{ day: number; activity: string }>
  keepActivities?: string[]                    // 客人明確要保留 → 受保護，不被自動移除
  // test 5 重排所需的結構化訊號（無訊號則不自動重排）：
  dayCapacities?: Array<{ day: number; maxStops: number }>
  activityAreaTags?: Array<{ activity: string; areaTag: string }>
}
```

### 逐條 decision 分類（純規則、共用 lint 的 token 表 = 單一事實來源）

- **test 1｜addActivities 含 mobility-unsuitable**（比對 `MOBILITY_UNSUITABLE_TOKENS`：叢林飛索/飛索/Jungle Flight/帕丘峽谷/帕丘/黏黏瀑布）
  → `declined`：不加進 draft；`retrievalCases` 有同主題替代景點就在 explanation 提名（白名單），沒有則泛稱不杜撰。
- **test 2｜finalDayMorningTransfer + 末日想吃午餐**（午餐落在末日）
  → `adjusted`：strip 末日午餐/晚餐/住宿；operatorNote「末日早上送機，餐宿已移除」，explanation 提「可改前一天安排」。
- **test 3｜sameLodgingAllTrip 但 lodgingChangeRequests 想插清萊/尼曼住宿**（住宿地名落在 `FORBIDDEN_LODGING_TOKENS`）
  → `declined`：不改住宿；explanation 建議「清萊改一日往返」。
- **test 4｜keepActivities 含天使瀑布**
  → `applied`：天使瀑布不在 unsuitable 表，原樣保留（共用 token 表 → 天然不誤殺，不靠特例）。
- **test 5｜過滿**（limited mobility 下某天 stop 數 > `dayCapacities.maxStops`）
  → `adjusted` 保守重排：把該天**末尾、且不在 keepActivities** 的活動，移到 stop 數最少的一天；
  **僅在有足夠結構化訊號時執行**（movable activity、keepActivities、dayCapacities、activityAreaTags 同區資訊）；
  **不跨區**（移入日的 areaTag 必須與該活動 areaTag 相符）；
  **無訊號 → 不自動重排**，只回 operatorNotes / customerExplanation，必要時 fail-closed；
  移完 render→lint 必須過，過不了則 fail-closed。
- **test 6｜變更與硬 constraint 衝突**：套完所有 decision 後 render→lint，殘餘 **error** → draft=null（warn 不擋）。
- **test 7｜customer-facing 乾淨**：draft 只含旅遊內容，調整理由全走 explanation / operatorNotes。

## Policy 確認（Eric 定稿）

- **declined** = 不寫入 draft，但給專業替代說法（走 explanation）。
- **adjusted** = 直接排成更舒服、可執行版。
- **applied** = 客人需求合理，原樣套用。
- render 後仍以 **lint error 作為最後 fail-closed gate**；warn 永不擋。
- 自動重排只在有足夠結構化訊號時執行；無訊號不亂跨天/跨區。

## 測試計畫（最低 regression，TDD）

1. 加飛索 + limited mobility → draft 不含飛索，回替代建議（declined）。
2. 末日早上送機 + 想吃午餐 → draft 不產末日午餐，explanation 提前一天（adjusted）。
3. 全程同住古城 + 插清萊住宿 → draft 不改住宿，explanation 建議一日往返（declined）。
4. 保留天使瀑布 → 原樣保留，不誤殺（applied）。
5. 已提供航班 → draft 不再寫「需確認航班」。
6. 改景點反映在 draft；與硬 constraint 衝突 → fail-closed（draft=null，仍回 explanation/operatorNotes）。
7. customer-facing draft 不輸出過多 internal/operator notes（禁詞檢查）。
8. test 5 過滿 + 足夠訊號 → 保守重排成功且過 lint；無訊號 → 不重排只回 notes。

## 共用點 / 單一事實來源

- 移除/替代/住宿衝突一律比對 `customer-itinerary-lint.ts` 既有 token 表
  （`MOBILITY_UNSUITABLE_TOKENS`、`FORBIDDEN_LODGING_TOKENS`），
  需從 lint 模組 export 這些表，composer 與 lint 永不打架。

## 實作狀態（2026-06-09 完成）

- 新檔 `src/lib/line-agent/notion/customer-itinerary-change-composer.ts`：
  `composeCustomerChange(input) → { ok, draft, customerExplanation, operatorNotes, issues }`。
- 三段流水線如設計：`applyChanges`（transform + decision 分類 applied/adjusted/declined）
  → 投影 explanation/operatorNotes → M3.3b `composeCustomerItineraryDraft` render+lint gate。
- `ok === (draft !== null)`；lint warn 不擋、error fail-closed；fail-closed 仍回 explanation/notes。

  `FORBIDDEN_LODGING_TOKENS`）＝單一事實來源，change 層與 lint 閘共用、永不打架。

- retrievalCases 僅白名單；無 fit 泛稱替代景點，絕不杜撰。
