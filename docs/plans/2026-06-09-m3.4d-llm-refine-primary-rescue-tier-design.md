# M3.4d — LLM Refine Primary→Rescue 兩層模型（設計）

> **For Claude:** REQUIRED SUB-SKILL: 用 superpowers:test-driven-development 逐 task RED→GREEN 實作本刀。

**Goal:** 在 M3.4b refine harness 上加「primary → rescue」兩層模型：primary（Haiku）候選被三道 guard 否決時，才升級給 rescue（Sonnet）再產一份候選過**同一條** guard；兩層皆 fail 才 fail-closed 退回 deterministic draft。並把 rejection reason `structural_drift` 正名為 `structural_diff`。

**Architecture:** 維持 M3.4b 的 seam 哲學——**注入點是 source、把關永遠是 deterministic code**。harness 不放寬任何 guard，只是讓第二個（更強）模型對**同一批 guard**再試一次。harness 仍 model-agnostic（只認 `source` / `rescueSource`，不認模型名）；模型名（haiku/sonnet）只活在 adapter/runner，harness 回傳 `tier`，runner 把 tier 對應成模型 label 印進 report。

**Tech Stack:** TypeScript（lib 純函式 + 可注入 source）、Node ESM runner（`.mjs`，唯一動態 import `@anthropic-ai/sdk`）、vitest、TDD。

## 2. 邊界

本刀**只能**：
- 在 harness 內加 `rescueSource` 第二注入點與 primary→rescue 編排；rescue 候選必過**同一條**三道 guard（lint / structuralDiff / leak），不放寬一字。
- adapter 加 rescue 模型解析常數/函式（對稱於既有 primary）。
- runner/command 組出 primary + rescue 兩個 source，report 加 tier 分佈（哪些 Haiku 收、哪些被 Sonnet 救回、哪些仍 fallback）。
- 把 `structural_drift` 正名為 `structural_diff`（harness rejectionReason + runner reason map + docstring）。

本刀**不**：不接 LINE、不翻 customer/partner gate、不寫 Sanity、不產正式 quote URL、不讓任一模型繞過 deterministic/lint guard、不自動跑真 API（沿用 M3.4c 修正 4：需 Eric 明確指令手動跑）。

硬邊界（report 禁止印出）：API key、token、DB id、Notion URL、成本、收入、利潤、客戶姓名、draft 全文、prompt 全文、internal/operator data。

## 4. harness：`customer-itinerary-refine.ts`

### 4.1 抽出 `evaluateRefineCandidate`（DRY）

把目前 inline 的三道 guard 跑法抽成純函式，primary / rescue 共用：

```ts
export interface RefineCandidateEvaluation {
  ok: boolean
  rejectionReasons: string[]       // lint_error | structural_diff | internal_leak
  structuralIssues: StructuralDiffIssue[]
  lintIssues: ItineraryLintIssue[]
  leakHits: string[]
}
export function evaluateRefineCandidate(
  deterministicDraft: string,
  candidate: string,
  constraints: CustomerItineraryConstraints
): RefineCandidateEvaluation
```

- 收集**全部** reason（不 short-circuit），與現況一致。
- `structural` 否決 reason **正名為 `structural_diff`**（取代 `structural_drift`）。

### 4.2 `rescueSource` + tier 編排

```ts
export interface RefineCustomerDraftInput {
  deterministicDraft: string
  constraints: CustomerItineraryConstraints
  source: RefineDraftSource            // primary
  rescueSource?: RefineDraftSource     // 新：選用；缺省＝維持單層（M3.4b 行為）
}

export interface RefineAttempt {
  tier: 'primary' | 'rescue'
  accepted: boolean
  rejectionReasons: string[]
  structuralIssues: StructuralDiffIssue[]
  lintIssues: ItineraryLintIssue[]
  leakHits: string[]
}

export interface RefineResult {
  draft: string
  deterministicDraft: string
  refinedDraft: string | null
  used: 'refined' | 'deterministic'
  tier: 'primary' | 'rescue' | null    // 新：採用的層；fallback → null
  attempts: RefineAttempt[]            // 新：跑過的層（1 或 2 筆）
  rejectionReasons: string[]
  structuralIssues: StructuralDiffIssue[]
  lintIssues: ItineraryLintIssue[]
  leakHits: string[]
}
```

**編排（fail-closed 不變）**：
1. primary source → 取候選。source throw → 記一筆 primary attempt（`rejectionReasons:['source_error']`）；空字串 → `['empty_output']`。否則 `evaluateRefineCandidate`。
2. primary `ok` → 採用，`used:'refined'`、`tier:'primary'`、`refinedDraft=primary 候選`、頂層 guard 欄位＝primary attempt（皆空）。
3. primary 被否決 **且** `rescueSource` 存在 → rescue source → 同步驟取候選 + evaluate，記第二筆 attempt。
   - rescue `ok` → 採用，`used:'refined'`、`tier:'rescue'`、頂層欄位＝rescue attempt（皆空）。
   - rescue 被否決 → fail-closed：`used:'deterministic'`、`tier:null`、`refinedDraft:null`、頂層欄位＝**最後一筆 attempt（rescue）**的 guard 資訊。
4. primary 被否決 **且** 無 `rescueSource` → fail-closed，頂層欄位＝primary attempt（**等同 M3.4b 行為**，僅 reason 名稱換成 `structural_diff`）。

**向後相容**：不傳 `rescueSource` 時，行為與 M3.4b 完全相同（`attempts` 長度 1、`tier` 為 primary 或 null）。既有 7 個 harness 測試只需把 `structural_drift` 斷言改 `structural_diff`（若有）即可，其餘綠不動。

## 5. runner：`scripts/refine-smoke-runner.mjs`

### 5.1 source 工廠改回傳 primary + rescue

`createSourceDefault` 由「回傳單一 source」改為「回傳 `{ primarySource, rescueSource }`」：同一個 `client`/`callModel`，用 `resolveRefineModel` 與 `resolveRescueRefineModel` 各綁一個模型，分別 `createAnthropicRefineSource({ apiKey, env, model, callModel })`。

- module **resolution** 失敗（SDK 未裝 / plain node 下 `.ts`）→ `null` → `factory_unavailable`（不打 API、不碰 key）。
- **construction** 失敗 → sanitized `RefineLlmWiringError`。
- 兩個 source 共用同一 client；rescue 只是換 model 字串。

`loadRefineLlmRuntime` 回傳擴充為 `{ status, reason, refineSource, rescueSource, kit }`；`kit` 加 `resolveRescueModel`。`importRefineKitDefault` 多撈 `resolveRescueRefineModel`。

### 5.2 report 加 tier 分佈

outcome 形狀：`{ caseId, model, rescueModel, promptLeak, result }`。

- `classifyRow`：採用 row 的顯示模型＝`result.tier === 'rescue' ? rescueModel : model`；fallback 且跑過 rescue（`result.attempts.length > 1`）→ 顯示 `model→rescueModel`。
- `summarizeRefineSmoke` 新增 `byTier: { primary, rescue, deterministic }`：primary 採用數 / rescue 救回數 / 仍 fallback 數。
- `formatRefineSmokeReport` 加一段「採用層分佈」：`primary（Haiku）：x · rescue（Sonnet）：y · 仍 fallback：z`。
- fallback reason / structural 子原因＝**最後一筆 attempt**（rescue 跑過就是 rescue）。現有 by-reason / structural breakdown / masking 全部沿用。
- reason map 正名：`REFINE_REASON_MAP` 的 `structural_drift` key 改 `structural_diff`（harness 已直接吐 `structural_diff`，映射成 identity）；註解的 `structural_drift` 字樣同步改。

### 5.3 command：`scripts/agent-command.mjs runRefineSmokeCommand`

- 取 runtime 時一併取 `rescueSource`（注入測試時可單獨給）。
- 每 case 過 prompt_leak 預檢後，呼叫 `kit.refine({ deterministicDraft, constraints, source, rescueSource })`。
- outcome 帶 `rescueModel = kit.resolveRescueModel?.({ env })`。
